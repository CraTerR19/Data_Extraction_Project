Wellness Atlas
Invoice Data Extraction Portal

Project Objective: A full-stack web application designed to securely accept user identification, upload medical utility PDF invoices, dynamically extract high-value text logic (Invoice Numbers, Dates, Amounts) using Optical Character Recognition (OCR), and seamlessly store this flow directly into a local MySQL database and AWS S3 object storage.

1. The End-to-End Workflow

The system operates in a precise, three-step multi-tier framework, connecting your browser flawlessly to powerful cloud capabilities and local persistence.

  • Frontend (UI/UX): Beautiful Glassmorphism interface capturing raw input. Users submit a unique User ID which gets securely bound to the DBeaver/MySQL instance.
  • File Transport (Upload): The user drops a PDF document using drag-and-drop mechanics. This is transported via FormData to the Python backend immediately securely.
  • Backend Extraction (OCR Services): The Python backend temporarily caches the PDF, safely uploads it to an AWS S3 Bucket via `Boto3`, and feeds the file into pdfplumber and pytesseract image-to-text parsers. Smart Regular Expressions hunt down key data points regardless of how unstructured the text is!
  • Persistence (Database): High-value tokens (Invoice Number, Date, Amount) are written deeply into the `invoices` table against the specific User ID via raw parametrised SQL executions.
  • Real-time Modification (Manual Overrides): The system bounces the parsed values back to the UI into perfectly styled text fields. If OCR has a strict failure, the user types changes, hits "Save", and the backend fires a PUT request, silently rewriting that specific row ID with absolute precision.

2. FastAPI Refactored Architecture

The backend was massively upgraded from standard Flask routing into a highly reliable FastAPI layered architecture (specifically inspired by industry-standard Python Enterprise design templates like `customer_service` patterns). By separating concerns, our application becomes highly testable, extremely decoupled, and readable.

core/ The Configuration Layer

This folder acts as the central hub for Environment definitions. The config.py securely exposes constants across the application globally (like AWS_ACCESS_KEY, DB_HOST). It completely removes hardcoded passwords from the actual operations, making it CI/CD ready.

db/ The Session Layer

session.py handles database hydration. Here we execute initialize_db() which builds our users and invoices schemas automatically, and we provide get_db_connection() generators so endpoints don't have to write raw PyMySQL connection loops.

services/ The Business Logic Layer

All OCR and Regex brainpower lives in ocr_service.py. By confining pdfplumber here, the API routes remain shockingly clean. Services represent "how" something is done (the brain), whereas APIs just ask the service for the answer.

api/ The Endpoints Layer

Nested deeply in api/v1/endpoints/invoices.py, we utilize FastAPI's APIRouter(). These functions merely validate incoming traffic from app.js, pass the traffic to the Services, grab the computed string mapping, and shoot back JSON.

3. Models vs Schemas (The Deep Dive)

In enterprise-grade FastAPI, specifically in SQLAlchemy/Pydantic ecosystems, we drastically separate Database representations from JSON representations. This duality is critical for security.

What are Models?

Models are pure representations of your database tables defined in python code (usually extending Base in SQLAlchemy). In Wellness Atlas, our model equivalents are the strict raw SQL tables defined directly in db/session.py (e.g. CREATE TABLE invoices (...)). Models dictate exclusively how data rests on the physical Disk/DBeaver schema.

What are Schemas?

Schemas (located in schemas/invoice.py) are Pydantic Data Guards representing how data travels over the Internet via JSON. They have strictly nothing to do with the disk!

class SubmitIdRequest(BaseModel): id: str # FastAPI will throw a 422 Error automatically if standard JSON lacks this key!

Why separate them? You never want users submitting a JSON payload directly into your DB Model (Model injection attacks). By forcing traffic through a Schema first, Pydantic strips out illegal variables magically. In our implementation, functions like submit_id(req: SubmitIdRequest) guarantee the input is perfectly typed before it ever touches `session.py`!
